# Part 1: Product Decisions

## 1. User Scenarios & Value Judgment

### 1.1 各角色邮件处理行为差异

| 角色 | 日均邮件量 | 邮件类型特征 | 现有工具覆盖 | 核心痛点 |
|------|-----------|-------------|-------------|---------|
| **IC（个人贡献者）** | 30–60封 | 项目更新、code review、会议通知 | 基本无 | 打断深度工作，但量不大，手动处理成本可控 |
| **Manager（中层管理者）** | 80–150封 | 审批、1:1跟进、跨团队协调、招聘、汇报 | 基本无 | 邮件量大、决策密度高、action item分散、无专属助理 |
| **Executive（高管）** | 100–200+封 | 战略决策、董事会、外部合作 | 多有EA辅助 | 需要信息提炼，但已有人力方案 |
| **Support（客服）** | 高但模板化 | 工单、SLA跟踪、升级 | Zendesk等ticketing系统 | 邮件只是通道之一，核心已被垂直工具覆盖 |
| **Sales（销售）** | 50–100封 | 客户跟进、报价、合同 | Salesforce/HubSpot等CRM | 时效性强但CRM已覆盖核心流程 |

### 1.2 核心用户：Tech Manager

V1的核心用户是**科技行业的中层管理者**，具体包括 Engineering Manager、Tech Lead、Technical Program Manager（TPM）。

**选择理由：**

**① 提效空间真实且可量化**
- 日均80–150封邮件，每天花费1.5–2小时处理，Agent有望将其压缩到30–45分钟
- 相比IC的30–60封（手动处理仅需30分钟），Manager的提效空间是IC的3–4倍

**② 痛点强烈且无替代方案**
- 不像Executive有EA辅助，Manager必须亲自处理每封邮件
- 不像Support/Sales有垂直工具覆盖，Manager的邮件场景缺乏专用工具
- Manager的邮件中action item高度分散——审批请求、会议跟进、下属问题、跨组协调——极易遗漏

**③ 技术背景降低V1风险**
- Tech Manager理解AI的能力边界，对V1的不完美容忍度高于非技术管理者
- 更愿意主动校准Agent的分类结果，形成有效的反馈循环
- 获客渠道明确：Hacker News、Dev Twitter/X、ProductHunt、技术社区

**④ 邮件模式在复杂度和规律性之间有良好平衡**
- 比IC的邮件复杂（涉及决策、委派），但比Executive的邮件结构化（PR review、sprint planning、incident有固定pattern）
- 模型能力和邮件复杂度的匹配度优于其他Manager类型

**不选其他角色的原因：**
- **IC**：日均邮件量不足，Agent的提效感知弱——"还不如自己翻一遍"
- **Executive**：多有EA辅助，市场规模小，且需要高度定制化
- **Support/Sales**：已有成熟垂直工具（Zendesk、Salesforce），邮件非核心工作界面
- **非技术Manager**：邮件内容更模糊、上下文更依赖组织政治，V1模型难以胜任

### 1.3 增长路径

```
V1  → Tech Manager（Engineering Manager, Tech Lead, TPM）
V1.5→ 非技术领域的中层管理者（Marketing Manager, Operations Manager）
V2  → 向上覆盖Executive（高级功能+定制化），向下扩散到高邮件量IC
```

## 2. MVP Trade-offs

### 2.1 资源约束

- 2名工程师 × 2个月 ≈ **4人月**
- 基础架构（邮件API接入、LLM pipeline、前端交互框架）预估 **1–1.5人月**
- 功能开发可用时间 ≈ **2.5–3人月**

### 2.2 各功能评估

| 功能 | 对Tech Manager的价值 | 技术复杂度 | 开发成本 | 风险等级 |
|------|---------------------|-----------|---------|---------|
| **(A) Priority Triage** | ★★★★★ 每天150封邮件的第一道防线 | 中 | ~3周 | 低——分错可被用户快速纠正 |
| **(B) Smart Reply Drafts** | ★★★★ Manager回复量大 | 高 | ~4周 | 高——语气/内容不当有社交风险 |
| **(C) Action Item Extraction** | ★★★★★ Manager最怕遗漏待办 | 中高 | ~3周 | 中——漏提取比错提取更危险 |
| **(D) Thread Summary** | ★★★★ 经常被拉入长thread | 低中 | ~2周 | 低——摘要不准用户可自行阅读原文 |
| **(E) Follow-up Reminders** | ★★★ 有价值但非差异化 | 中 | ~2.5周 | 低——已有竞品方案（Superhuman等） |

### 2.3 MVP方案：纳入 A + C + D，排除 B + E

#### ✅ Include

**(A) Priority Triage — 核心入口（~3周）**

将收件箱邮件分类为"Act Now / Later / Ignore"三个层级。

- 这是整个Agent的**基础能力**，后续所有功能依赖对邮件的理解和分类
- 直接解决Tech Manager最高频的痛点："打开150封邮件，我应该先看哪几封？"
- 分类所用的信号（发件人、主题、内容、上下文）为C和D提供共享的分析pipeline
- 分错可被用户一键纠正，且纠正数据反哺模型优化

**(C) Action Item Extraction — 核心价值（~3周）**

从邮件中提取待办事项，生成to-do或日历事件建议（用户确认后执行）。

- 直击Tech Manager核心痛点：审批请求、会议跟进、文档更新散落在数十封邮件中
- 与A高度协同——"Act Now"邮件自动提取action item，形成**分类→提取→执行**的完整闭环
- 复用A的邮件解析pipeline，边际开发成本低
- 提取结果以建议形式呈现，用户确认后才写入to-do/calendar，控制风险

**(D) Thread Summary — 低成本高感知（~2周）**

对长邮件链生成结构化摘要。

- Tech Manager经常被中途拉入20+封的讨论thread，花15分钟读完才发现只需回一句话
- LLM summarization是最成熟的能力之一，开发成本最低，但用户感知价值高
- 与A配合：先看优先级分类，再对重要的长thread一键摘要，大幅缩短处理时间

**三个功能的协同关系：**

```
Tech Manager打开收件箱（150封邮件）
    │
    ▼
(A) Priority Triage
    "12封Act Now / 35封Later / 103封Ignore"
    │
    ▼  点开一封20+封的Act Now thread
(D) Thread Summary
    "讨论的是Q2 launch timeline变更，最新结论是推迟到6月，需要你审批新timeline"
    │
    ▼  确认理解后
(C) Action Item Extraction
    "→ To-do: 审批Q2 launch新timeline, Due: Today"
    "→ To-do: 通知设计团队更新交付日期, Due: Tomorrow"
    │
    ▼  用户确认 → 写入To-do / Calendar
    处理完成，进入下一封
```

**形成"感知→理解→行动"的完整工作流。**

#### ❌ Exclude

**(B) Smart Reply Drafts — 延后到V2**

- **风险过高**：Manager的回复涉及审批、委派、跨组协调，语气和内容的错误可能造成社交损害（如用错误的口吻回复VP，或在敏感话题上给出不恰当的回应）
- **质量阈值高**：生成回复需要深度理解用户的写作风格、组织层级关系、项目上下文，V1数据积累不足
- **开发成本最高**（~4周），占总预算的40%+，排挤其他功能
- **V2策略**：在V1积累用户交互数据后，基于真实的回复pattern训练，质量更有保障

**(E) Follow-up Reminders — 延后到V1.5**

- 有价值但**非差异化能力**——Superhuman、Boomerang等已有成熟方案
- Tech Manager主动跟进的场景虽然存在，但频率低于A/C/D覆盖的场景
- 资源应优先投入竞品做不好的agentic能力（理解上下文+提取行动项），而非追赶已有feature
- 技术上相对独立，适合作为V1.5的增量功能快速补充

## 3. Risk Judgment

### 3.1 两类核心错误的危险性对比

| 错误类型 | 场景举例 | 后果严重性 | 发生概率 |
|---------|---------|-----------|---------|
| **False Negative（重要邮件被标为Ignore）** | VP的紧急审批请求被归入Ignore | ⚠️ **极高** — 可能错过关键决策窗口，影响项目进度甚至职业关系 | 中——对陌生发件人或非典型主题容易误判 |
| **False Positive（普通邮件被标为Act Now）** | 全员通知邮件被标为Act Now | 低——用户多花几秒看一眼就能dismiss | 高——模型倾向保守时会过度标记 |

**结论：False Negative 远比 False Positive 危险。**

一封被错误忽略的重要邮件可能导致不可逆的后果（错过deadline、遗漏VP指示），而一封被错误提升的普通邮件只浪费用户几秒钟。

### 3.2 产品设计层面的风险控制

**原则：宁可多提醒，不可漏提醒（Bias toward False Positive）**

| 策略 | 具体做法 |
|------|---------|
| **分类阈值非对称设计** | 将"Ignore"的置信度阈值设为90%+，而"Act Now"的阈值设为60%即可触发。宁可Act Now多一些，也不让重要邮件漏到Ignore |
| **Ignore不等于删除** | Ignore类邮件仍然可见，只是折叠展示。用户随时可以展开检查，发现误判后一键纠正 |
| **高风险发件人白名单** | 用户可设置"永远不Ignore"的发件人列表（如直属上级、VP、关键客户），这些发件人的邮件至少归入Later |
| **渐进式信任建立** | 第一周只展示分类建议，不折叠任何邮件；用户确认准确率达标后，逐步启用自动折叠Ignore邮件 |
| **每日Ignore摘要** | 每天下班前推送一份Ignore邮件的简要摘要，作为safety net让用户快速扫描是否有遗漏 |
| **所有action均需确认** | MVP阶段，Agent的所有执行操作（创建to-do、添加日历、标记flag）都需用户明确确认，不做静默自动化 |

### 3.3 Action Item提取的特殊风险

| 风险 | 危害 | 缓解措施 |
|------|------|---------|
| 漏提取关键action item | Manager遗漏重要待办 | 对Act Now邮件强制展示原文关键段落，不完全依赖提取结果 |
| 错误提取（将讨论性内容识别为action item） | 产生无效to-do，用户烦躁 | 对每个提取结果标注置信度和原文出处，低置信度的以"可能的待办"形式弱化展示 |
| 重复提取（同一thread多封邮件提取相同action） | to-do列表冗余 | 基于thread ID做去重，同一action item只保留最新版本 |

## 4. Kill Criteria

以下任一条件成立时，应建议**不构建或延后**此产品：

### 4.1 技术层面

| Kill条件 | 判断标准 | 原因 |
|---------|---------|------|
| **邮件API接入受阻** | 在2周内无法完成Gmail/Outlook API的稳定接入和权限获取 | 无法读取邮件，整个产品无基础可言 |
| **Priority Triage准确率不达标** | 内测阶段Act Now/Later/Ignore的分类准确率持续低于75% | 低于75%意味着用户每4封邮件就需要纠正1封，净收益可能为负 |
| **LLM延迟不可接受** | 单封邮件的分类+摘要响应时间超过5秒，且无法通过缓存/批处理优化 | 用户不会等5秒看一封邮件的分类结果——比自己看还慢 |

### 4.2 产品层面

| Kill条件 | 判断标准 | 原因 |
|---------|---------|------|
| **用户信任无法建立** | 内测用户连续2周后，仍有>50%的人手动检查Ignore分类 | 说明用户不信任Agent的判断，产品沦为"多了一层需要检查的东西" |
| **隐私合规风险** | 企业客户的安全团队在POC阶段明确拒绝将邮件内容发送到外部LLM | 没有数据就没有产品，而自建模型超出2工程师的能力范围 |
| **使用频率持续走低** | 用户在第一周后的DAU周留存率低于30% | 说明痛点不够强，或产品体验不足以改变用户的邮件处理习惯 |

### 4.3 市场层面

| Kill条件 | 判断标准 | 原因 |
|---------|---------|------|
| **平台方原生能力追上** | Microsoft Copilot或Google Gemini在6个月内发布了同等能力的邮件分类功能 | 作为第三方插件，在平台方原生能力面前几乎没有竞争优势 |
| **付费意愿验证失败** | 50个目标用户访谈中，愿意为此功能付费（>$10/月）的比例低于20% | 说明痛点虽然存在但不够强烈，无法支撑独立产品的商业模式 |
